这篇写一下我平时写代码时，Cursor 和 Openclaw 是怎么配合用的。不是教程，就是真实用法和感受。

### 场景：我平时在干什么

我非科班出身，写代码多半是查文档、改配置、修报错。以前全靠搜索引擎 + 复制粘贴，现在会先把问题丢给 AI，再在编辑器里改。Cursor 负责「在项目里改代码」，Openclaw 负责「在聊天/命令行里问问题、跑脚本、看邮件」，两边各干各的，但会一起用。

### Cursor 在我这的用法

- **改小地方**：改一两行配置、修个报错、补个函数，直接选中代码让 Cursor 改，省得自己查 API。
- **起手新文件**：新建一个脚本或配置，用自然语言说「帮我写一个 xxx」，它按项目上下文生成，我再微调。
- **看不懂的报错**：整段贴进去，让它解释是哪里出问题、该怎么改。

我不指望它一次写对一整块业务逻辑，但做「填空」和「改错」很省时间。

### Openclaw 在我这的用法

- **在 QQ / 命令行里问问题**：不在 Cursor 里的时候，就用 Openclaw 接的模型问，比如「这段日志什么意思」「这个命令怎么查」。
- **看邮件、做小结**：邮件摘要、待办提炼这类，用 Openclaw 跑脚本或发 prompt，和写代码是分开的一条线。
- **树莓派 / 服务器**：Openclaw 在远程也跑着一份，所以不在本机也能用同一套模型。

也就是说：**写代码时用 Cursor，不写代码时的问答和自动化用 Openclaw**。

### 两边配合时的问题

1. **上下文不互通**：Cursor 不知道 Openclaw 里聊过啥，Openclaw 也不知道我当前文件内容。要跨工具时只能自己复制粘贴、说清楚背景。
2. **模型不一致**：Cursor 用自家的模型，Openclaw 用我配的国内 API。有时候同一个问题两边回答风格不一样，我会以「谁这次说得更靠谱」为准，没有固定规则。
3. **成本**：Cursor 订阅要钱，Openclaw 这边用免费额度时要注意 token 和 fallback，不然容易误跑贵模型。这部分和我之前写的 Openclaw 填坑记一样，要自己限好重试和用量。

### 小结

对我来讲，这个组合的意义就是：**写代码有 Cursor 兜着，其它杂事有 Openclaw 兜着**。没有追求「全在同一个界面」，够用、稳定、别乱烧钱就行。如果你也在用 Cursor + 自建/开源 AI，可以按自己的习惯拆成「编辑器中」和「编辑器外」两条线，会清晰很多。
